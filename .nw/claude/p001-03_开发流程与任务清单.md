## 一、开发流程：「垂直切片 → 水平扩展」

### 概念定义

**水平分层开发**（传统方式）是将应用按技术层拆分——先做完所有 API 层，再做所有 Store 层，再做所有 UI 组件，最后拼页面。这种方式的问题是：每一层完成后无法独立验证——API 层写完了但没页面调用，不知道对不对；UI 组件写完了但没真实数据，不知道够不够。直到最后一层页面组装时才暴露问题。

**垂直切片开发**相反：选取一个完整功能（如"用户管理"），从最顶层的页面视图一直切到最底层的 HTTP 请求，打通应用的全部技术层。这个切片本身就是架构的**可运行验证**——如果用户管理能跑通，说明 Vite 配置、Axios 拦截器、OIDC 流程、Pinia store、Element Plus 集成、DataTable/Modal/DynamicForm 组件全部正确。

**水平扩展**是指：垂直切片验证通过后，剩余功能不再需要"探索模式"。每个新页面遵循切片中已定型的代码模式——复制 UserCreateEditModal 的结构，把 API 换成 identity-roles，把字段换成 name/isDefault/isPublic，角色管理就完成了。这是**确定性工作**，因此可以全部并行且几乎不会返工。

### 为什么本项目特别适合

ABP 管理页面的**结构同构性**是这种策略成立的前提。7 个管理页面（用户/角色/租户/权限/功能/邮件/时区）共享完全相同的骨架：

```
列表页 = <AbpDataTable :columns="cols" :api="getList" :on-delete="delete">
模态框 = <AbpModal :visible="show" @confirm="submit">
           <AbpDynamicForm :fields="fields" :schema="zodSchema" />
         </AbpModal>
行操作 = [编辑按钮] [删除按钮] [权限/功能管理入口按钮]
```

差异仅在于三个参数：**columns**（表格列定义）、**fields**（表单字段列表）、**api**（调哪个端点）。这意味着在 A7 中定型一个模式后，B1-B6 每个任务的核心工作量从"设计+实现+调试"降级为"配置三个参数"。这正是 Phase B 可以全并行且单任务只需 0.5-1d 的原因。

### 具体模式复用示例

A7 中实现的 `UserCreateEditModal` 包含以下全部能力：
- 打开/关闭动画 + 焦点管理
- VeeValidate 脏检查 → 关闭前"未保存"确认
- 提交中按钮 loading + 禁用
- 服务端 RemoteServiceErrorResponse → 字段级错误映射
- 并发戳冲突提示

B1 的 `RoleCreateEditModal` **不需要重新实现任何一项**。它只需：
```ts
// 差异部分
const fields = [
  { type: 'text', name: 'name', label: 'RoleName', required: true },
  { type: 'switch', name: 'isDefault', label: 'IsDefault' },
  { type: 'switch', name: 'isPublic', label: 'IsPublic' },
]
const api = { create: identityRolesApi.create, update: identityRolesApi.update }
// 复用部分（零改动）
<AbpModal ...><AbpDynamicForm :fields="fields" :schema="toTypedSchema(zodSchema)" /></AbpModal>
```

这就是"水平扩展"的本质：Phase A 投入 1.5d 写出用户管理，Phase B 每个模块投入 0.5-1d 做字段配置 + API 替换，不再触碰基础设施和 UI 组件逻辑。

### 与纯并行（合约先行）的对比

| 维度 | 纯并行（合约先行） | 垂直切片 → 水平扩展 |
|------|------------------|-------------------|
| 架构风险 | 并行 12 模块，集成时发现问题全部返工 | Phase A 一条链路暴露全部架构问题 |
| 模式复用 | 各模块独立摸索，UI 不一致 | A7 定型模式被 Phase B 100% 复用 |
| 进度可见 | 无一可运行 | A7 完成即有可演示的登录+用户管理 |
| Phase B 并行度 | 全部可并行 | 全部可并行（与合约模式相同） |
| 实际效率 | 理论高（返工抵消） | 更高（模式复用削减 B 任务至 0.5-1d） |

### 整体架构

```
Phase A: 垂直切片（P0，顺序）           Phase B: 水平扩展（P1-P3，全部可并行）

A1 脚手架                                B1 [P1] 角色管理 ─────────────┐
  ↓                                       B2 [P1] 个人资料+认证页 ──────┤
A2 API+HTTP+类型                           B3 [P2] 租户管理 ─────────────┤
  ↓                                       B4 [P2] 权限管理(对象式) ─────┤ 全并行
A3 Pinia 状态层                            B5 [P2] 功能管理(对象式) ─────┤
  ↓                                       B6 [P3] 设置管理 ─────────────┤
A4 i18n+权限+OIDC                         B7 [P3] 集成收尾 ─────────────┘
  ↓
A5 UI 组件定型（DataTable/Modal/Form）     ← 全部依赖 A7 的已验证模式
  ↓
A6 布局系统
  ↓
A7 登录+用户管理（端到端验证）
  ↓
✅ 架构验证 → 启动 Phase B
```

## 二、Phase A — 垂直切片（P0，顺序）

### A1 [P0] 项目脚手架（0.5d）
- **内容**：Vite + Vue3 + TS + Element Plus（unplugin-vue-components 按需导入）+ Sass（`additionalData`）+ Tailwind CSS（断点 576/768/992/1200）+ 空路由 + 入口文件
- **验收**：`npm run dev` 正常启动

### A2 [P0] API 层 + HTTP + 类型（1d）
- **数据来源**：`docs/swagger.json`（57 端点，DTO 类型直接提取）
- **内容**：`src/types/`（全部 DTO）+ `src/api/http.ts`（Axios + 5 拦截器）+ `src/api/*.ts`（7 模块纯函数）

### A3 [P0] Pinia 状态层（0.5d）
- **内容**：`app-config`（启动时 fetch 配置）/ `auth`（user+token，持久化）/ `session`（语言+租户，持久化）+ `abp-error.ts`（RemoteServiceErrorResponse 解析）

### A4 [P0] i18n + 权限 + OIDC（1d）
- **内容**：`plugins/i18n.ts`（vue-i18n+ABP 远程加载）/ `composables/usePermission.ts` + `v-permission` + 权限守卫 / `plugins/oidc.ts`（PKCE+静默刷新+Remember Me）+ 认证守卫

### A5 [P0] UI 组件定型（2d）
- **内容**：`AbpDataTable`（基于 el-table：分页/排序/搜索/行操作/空状态/骨架/列可见性/响应式）/ `AbpModal`（基于 el-dialog：多尺寸/脏检查/提交加载/错误映射/全屏）/ `AbpDynamicForm`（配置驱动，首批 8 字段类型，VeeValidate+Zod）/ `AbpToast`/`AbpLoaderBar`/`AbpBreadcrumb`/`AbpEmptyState`/`AbpConfirmDialog`

### A6 [P0] 布局系统（1d）
- **内容**：ApplicationLayout（侧边菜单+顶栏+面包屑）/ AccountLayout（居中认证框）/ EmptyLayout + 子组件（SideMenu/TopBar/UserMenu/LangSwitch/TenantBox）

### A7 [P0] 登录页 + 用户管理（1.5d）
- **内容**：LoginView（用户名/密码/记住我/租户切换/密码可见性/Caps Lock）/ UsersView（分页+搜索+新建/编辑/删除）/ UserCreateEditModal（动态表单+角色分配标签页）
- **端到端验证**：OIDC 登录 → 用户列表 → 新建 → 编辑 → 删除
- ✅ **A7 完成 = Phase B 启动条件满足**

## 三、Phase B — 水平扩展（P1-P3，全部并行）

> 每个任务模式：参考 A7 的 DataTable+Modal+Form → 替换 API → 调整字段。

### B1 [P1] 角色管理（1d）
分页表格 + 新建/编辑模态框（name/isDefault/isPublic）。静态角色徽章（不可删除）、默认角色徽章。权限入口：`openPermissionModal({ providerName: 'R', providerKey: roleName })`。API：`identity-roles.ts`（6 端点）。

### B2 [P1] 个人资料页 + 剩余认证页面（1d）
RegisterView / ForgotPasswordView / ResetPasswordView / ManageProfileView。API：`account.ts`。LoginView 已在 A7 完成。

### B3 [P2] 租户管理（1d）
分页表格 + 新建/编辑模态框（创建含密码，编辑无密码）+ 连接字符串面板 + 启用/禁用开关。功能入口：`openFeatureModal({ providerName: 'T', providerKey: tenantId })`。API：`tenant.ts`（7 端点）。

### B4 [P2] 权限管理 — 对象式 API（1.5d）
`openPermissionModal({ providerName, providerKey })` → `{ open(), close(), onSaved() }`。UI：权限树（按组标签页）+ 级联选择 + 授权来源徽章 + 变更摘要 + 资源权限（列表/添加/编辑）。API：`permission.ts`（8 端点）。

### B5 [P2] 功能管理 — 对象式 API（1d）
`openFeatureModal({ providerName, providerKey })` → `{ open(), close(), onSaved() }`。UI：按组 + 3 值类型（Toggle/FreeText/Selection）+ 级联 + 变更摘要。API：`feature.ts`（3 端点）。

### B6 [P3] 设置管理（1d）
标签页容器 + 邮件设置（SMTP + 测试邮件）+ 时区设置。API：`settings.ts`（6 端点）。

### B7 [P3] 集成收尾（1d）
403/404/500 错误页路由 + 全局离线/超时 + 响应式验证（768px）+ `<title>` 动态设置 + 审计字段 + 对照 S006 验证。
