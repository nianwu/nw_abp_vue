# Skill: 项目实施规划

## 触发条件
当用户要求制定开发计划、实施规划、任务拆分、模块边界设计时。

## 流程

### 1. 依赖调研
按全局 CLAUDE.md「第三方库选型规则」：识别需求 → 搜索候选 → 四维度 + TS 集成质量评估 → 输出对比表（★☆评分、淘汰理由、最终推荐）。每个关注点最多推荐三个候选。

### 2. 模块边界
对每个模块定义三层合约：
- **拥有**：该模块独占的逻辑/资源
- **导出**：对外暴露的接口（函数签名/组件 Props/Store getter）
- **不拥有**：明确不属于该模块的职责

### 3. 目录结构
遵循目标框架社区常规命名（Vue: `views/` `components/` `composables/` `stores/` `api/` `plugins/` `layouts/`；React: `pages/` `components/` `hooks/` `stores/` `services/`）。

### 4. 开发流程
采用「垂直切片 → 水平扩展」：

**Phase A — 垂直切片（顺序执行）**：
选取一个完整功能，从页面视图直切到 HTTP 请求，打通全部技术层。目的是**在最短时间内验证全栈架构**。垂直切片完成即意味着所有基础设施（构建/HTTP/状态/认证/UI组件/布局）已验证可运行。

**Phase B — 水平扩展（全部并行）**：
基于 Phase A 定型的模式，按优先级批量扩展剩余功能。每个功能只需替换差异部分（字段配置、API、列定义），复用全部通用逻辑。所有任务可并行且几乎不返工。

**适用条件**：项目的功能模块共享高度一致的结构骨架（如同为 CRUD 页面、同为表单页面）。

### 5. 任务清单
- 每个任务标注优先级（P0 基石 → P1 核心 → P2 扩展 → P3 辅助）
- 标注依赖关系（A 依赖 B、可并行）
- 标注预估工时
- 标注第三方依赖及其许可证

### 6. 验收体系（三层）
- **任务自查**：每个任务有独立的自查清单（含共享关注项如响应式/空状态/错误路径，确保不遗漏）
- **阶段门禁**：在关键节点设置不可跳过的验证条件（如 Phase A → B 须验证完整链路可走通、UI 模式可复用）
- **整体验收**：对照需求清单 + API 契约 + 构建检查 + 许可证审查

### 7. 计划文件
- 写入 `.nw/claude/p00x_【目的】.md`，`p00x` 递增
- 目的发生实质变化时创建新文件（递增）而非覆盖
- 文件必须含「修订记录」（`YYYY-MM-DD HH:mm:ss`）和「实施记录」
- 单个 .md ≤ 200 行；超标则按逻辑模块拆分子文件，以 `p00x-NN_【子主题】.md` 命名，主文件通过 `[链接](./子文件.md)` 索引

## 输出格式

```
# p00x — 【目的】（总览）

→ 全部任务细节见 [p00x-NN](./p00x-NN.md)
```
